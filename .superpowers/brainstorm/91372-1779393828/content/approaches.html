Three Approaches: digit_number Component

How to sample segments and classify digits. Pick the architecture that fits your constraints.

A

Single-point sampling per segment

User defines one (x,y) coordinate per segment (7 × 4 = 28 points). Component reads brightness at that exact pixel. If brightness > threshold → segment ON. Decode pattern → digit.

Pros

  • Simplest config
  • Minimal CPU on ESP32
  • Deterministic

Cons

  • Sensitive to blur / camera shift
  • One bad pixel = wrong digit
B

Area sampling per segment ⭐ Recommended

User defines one (x,y) center + optional radius (default 2px). Component averages brightness over NxN patch. Works despite blur. Threshold fixed or auto (auto = half-way between darkest and brightest patch seen this frame).

Pros

  • Blur-tolerant (glow spreads → still bright)
  • Still cheap on ESP32 (small patch)
  • Auto-threshold adapts to lighting
  • Camera can be moved closer

Cons

  • 28 center coords still needed in YAML
  • Slightly more config than A
C

Digit bounding box + auto segment layout

User defines bounding box per digit (x,y,w,h). Component divides the box into 7 standard 7-segment zones automatically. No per-segment coords needed.

Pros

  • Minimal YAML config (4 boxes vs 28 points)
  • Easier to set up

Cons

  • Less precise — assumes standard segment proportions
  • Fails if digits aren't standard 7-seg layout
  • Harder to debug misreads
